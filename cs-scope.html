What It Does

Scope XY-7 is a macOS menubar app that captures system audio and renders it as a real-time Lissajous (XY) or waveform display. It simulates the look of an analog CRT oscilloscope — phosphor persistence, bloom glow, vignette darkening — all running at 60fps in a borderless floating window.

No virtual audio drivers. No kernel extensions. Just ScreenCaptureKit piping system audio through a lock-free ring buffer into a Core Graphics offscreen renderer.

Key Features

  • XY (Lissajous) and waveform display modes
  • Adjustable phosphor persistence and gain
  • 4 phosphor colors: green, amber, white, orange
  • CRT bloom glow via downscale/upscale compositing
  • Light and dark mode
  • PNG snapshot and MP4 video recording
  • Floating borderless window with dynamic dock icon
  • Auto-relaunch permission flow for new users

Why Build This

I wanted to see what music looks like. Lissajous figures — where left audio maps to X and right maps to Y — turn stereo separation, phase relationships, and frequency ratios into visual geometry. A mono signal is a diagonal line. A pure stereo sine wave is a circle. Real music produces constantly evolving organic shapes.

Existing tools were either expensive hardware, Windows-only, or web apps that couldn't capture system audio without browser hacks. macOS had no native option that was lightweight, beautiful, and just worked from the menubar.

Architecture

The app has four layers, each running on its own thread or context:

  • Audio capture — ScreenCaptureKit SCStream on a high-priority GCD queue, extracting interleaved float samples from CMSampleBuffer
  • Ring buffer — 2048-sample circular buffer with NSLock. Snapshots are copied in-place so the renderer never blocks the audio thread
  • Renderer — Core Graphics offscreen CGContext. Persistence is achieved by partially clearing the buffer each frame (filling with background color at low alpha). Grid and vignette are cached as CGImages and only rebuilt on size/color/mode change
  • Display — NSView driven by CADisplayLink at 60fps. The view reads directly from the ring buffer snapshot — no main thread dispatch for audio data

This separation means the audio callback never touches the main thread, the renderer never allocates, and SwiftUI controls stay responsive even during heavy rendering.

Rendering Pipeline

Each frame follows this sequence:

  • Fade — Fill the persistence buffer with the background color at (1 - persistence) alpha. This creates the phosphor decay trail.
  • Trace — Draw the audio path as two strokes: a wide dim outer (4px, 25% alpha) for phosphor spread, then a sharp bright core (1.5px, 100%) on top
  • Bloom — Downscale the trace buffer to 25%, then draw it back at full size. Bilinear upscaling acts as a free Gaussian blur, creating the CRT glow
  • Grid — Composite the cached grid overlay (8x8 divisions, center crosshair, diagonal guides in XY mode)
  • Vignette — Composite the cached radial gradient for edge darkening

The trace is downsampled to ~512 points per frame (from 2048 in the ring buffer) for performance. At 60fps on a 420px display, this is well above visual fidelity requirements.

Design Decisions

Teenage Engineering language

The UI uses 9px monospaced type, 1px borders, flat controls with no shadows, and #FF6600 orange as the only accent color. This creates a tool aesthetic — the interface disappears and the visualization is the product.

Borderless floating window

A popover felt too disposable for something you want to keep visible while working. A full window felt too heavy for a utility. The floating panel splits the difference: always on top, draggable, resizable, but no title bar chrome. The dock icon only appears when the window is open.

Permission flow

ScreenCaptureKit requires Screen Recording permission, which is confusing for an audio-only app. The onboarding screen explains this upfront, auto-detects when permission is granted, and relaunches the app automatically. The user never has to manually quit and reopen.

Challenges

Main thread starvation

The first version dispatched 4096 float samples to the main thread on every audio callback (~86 times per second). This choked SwiftUI and made buttons unresponsive. The fix: store snapshots directly in the ring buffer and have the renderer read them on its own schedule. Only a single boolean (hasSignal) crosses to the main thread.

TCC permission loop

During development, Xcode re-signs the binary on every build, which invalidates the Screen Recording permission grant. This created a frustrating loop of granting, building, losing permission. The solution for development: sign with a stable team identity. For users: the auto-relaunch flow handles the one-time grant seamlessly.

CVDisplayLink deprecation

The original frame timer used CVDisplayLink, which Apple deprecated. Migrated to CADisplayLink, but the macOS API differs from iOS — you create it via the NSView instance method, not the class initializer.

Reflection

This was my first native macOS app and my first time working with real-time audio and offscreen rendering. The biggest lesson: performance problems are architecture problems. Moving data between threads is where frame budgets die, not in the rendering math itself.

The Teenage Engineering design constraint was freeing — with only one accent color and monospaced type, every decision was about information density and visual hierarchy rather than aesthetic choices.

Next: signed notarized distribution via DMG, Sparkle auto-updates, and an FFT spectrum mode.
