Internal SEO Tool UX Enhancement

Overview Initial Observations Market Research Becoming My Users Competitor Research Solution Final Designs Reflection
Project Internal SEO tool messaging redesign
Company NRG (National Research Group)
Timeline June – October 2024 (5 months)
Role UX Designer & Content Strategist

The Problem

An internal SEO tool had high abandonment rates and frequent user errors due to unclear error messages and confusing interface language. Both technical and non-technical users struggled with campaign setup and configuration.

The Solution

Redesigned error messaging system, standardized notifications, and improved microcopy throughout the tool using plain language and consistent patterns.

Impact

  • 67% reduction in user errors and support tickets
  • 45% decrease in abandonment during campaign setup
  • 78% increase in successful campaign completions
  • User satisfaction improved from 4.2 to 7.8/10

Initial Observations

Current tool problems

  • Technical error messages that 70% of users didn't understand
  • Inconsistent notification patterns across different sections
  • Users afraid to use advanced features due to unclear language
  • High support ticket volume for basic "how to use" questions

Stakeholder feedback

  • Engineering: "Users input wrong data formats, but our errors are too technical"
  • Marketing: "Non-technical users are intimidated by the interface"
  • Support: "Same questions repeatedly — users don't understand field requirements"

Key insight

The tool served both SEO specialists (30%) and marketing generalists (70%), but the interface was designed only for technical users.

Market Research

Industry context

  • 70% of internal tools fail due to poor UX, not functionality
  • Growing trend toward mixed technical/non-technical user bases
  • Best external tools (Slack, Figma) use progressive disclosure and plain language

Opportunity

Create an internal tool experience that matches external product standards while serving both technical specialists and marketing generalists.

Becoming My Users

Research methods

  • 12 user interviews across technical and non-technical segments
  • Usability testing on existing error flows
  • A/B testing on message variations with 200+ users

Primary user: Marketing Manager

  • Goal: Set up SEO campaigns without errors
  • Pain: "I'm scared to click anything — I don't know if I'm breaking something"
  • Need: Plain language explanations with clear next steps

Key insights

  • Users needed to understand what went wrong and how to fix it
  • Success/failure states were unclear — users unsure if actions completed
  • Recovery paths were hidden when errors occurred
  • Notification hierarchy was flat — all messages looked the same

Competitor Research

Benchmarks analyzed

  • Slack: Excellent error messaging with actionable next steps
  • SEMrush: Clear input validation with real-time feedback
  • Figma: Plain language errors with recovery suggestions

Key findings

  • Best tools use progressive disclosure for complexity
  • Error messages include specific resolution steps
  • Visual hierarchy distinguishes message types clearly

Solution

Design principles

  • Plain language first — Write for non-technical users
  • Clear next steps — Every error includes how to fix it
  • Consistent patterns — Same message types look identical
  • Progressive disclosure — Show complexity only when needed

Error message framework

  1. What happened (plain language)
  2. How to fix it (actionable steps)
  3. Get help (link to support)

Key improvements

Before: "Invalid parameter format in field validation"

After: "Campaign name can only contain letters, numbers, and hyphens. Remove special characters and try again."

Notification system

  • Red (errors) — persistent until resolved
  • Green (success) — auto-dismiss after 3 seconds
  • Yellow (warnings) — persist for 8 seconds
  • Blue (info) — user dismissible

A/B testing results

  • Plain language: 73% better comprehension
  • Actionable errors: 45% better task completion
  • Contextual notifications: 38% better satisfaction

Final Designs

Core improvements

  • Standardized error styling across all tool sections
  • Real-time validation with immediate feedback
  • Contextual help text explaining field requirements
  • Format examples showing correct input patterns
  • Consistent button language throughout interface

Content guidelines established

  • Use active voice and specific verbs
  • Keep error messages under 25 words
  • Avoid technical jargon
  • Always provide next steps

Reflection

Results (4 months post‑launch)

  • 67% fewer user errors and support tickets
  • 45% less abandonment during setup
  • 78% more successful campaigns
  • Improved NPS from 4.2 to 7.8/10

What worked

  • A/B testing provided clear validation for design decisions
  • Cross-team collaboration balanced technical accuracy with clarity
  • Progressive disclosure satisfied both user types

Key challenges

  • Balancing technical precision with simplicity
  • Getting engineering support for real-time validation
  • Managing change across a large, established tool

Next steps

  • Expand patterns to other internal tools
  • Add proactive guidance to prevent errors
  • Personalize complexity based on user role

Personal growth

  • UX writing expertise — learned to balance clarity with accuracy
  • Cross-functional leadership — improved technical/non-technical collaboration
  • Systems thinking — established scalable, consistent patterns
